Introduction

Background & Strategic Objectives

To resolve persistent pain points including double pipeline runs on Merge Requests, fragile artifact retention, and spotty JaCoCo test visibility, we designed a comprehensive monorepo restructuring. By containerizing independent steps into self-contained sub-stages, we reduced build noise while reinforcing pipeline speed and scale.

Double Pipelines Fixed

Prevented both branch and MR pipelines from triggering for identical commits.

Centralized Blueprint

Leveraged spec:inputs and modular reference configs to cleanly reuse configurations.

Optimal Performance

Leveraged FastZip & tiered JVM optimization flags to decrease runtimes.

Root Config

Root Orchestrator Configuration

The core router (`.gitlab-ci.yml`) determines the stage sequence, defines trigger criteria, and filters target paths to avoid running tests for unchanged directories. 

workflow:
  rules:
    - if: $CI_FULL_PIPELINE == "true"
    - if: $CI_PIPELINE_SOURCE == "parent_pipeline"
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH && $CI_OPEN_MERGE_REQUESTS # Suppress double pipeline runs
      when: never
    - if: $CI_COMMIT_BRANCH

stages:
  - configuration
  - foundation
  - application
  - test
  - report

.trigger-rules:
  rules:
    - if: "$CI_FULL_PIPELINE == 'true'"
      when: always
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
      changes:
        compare_to: $CI_MERGE_REQUEST_TARGET_BRANCH_NAME
        paths:
          - $MODULE_PATH/**/*
          - parent-pom/**/*
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH && $CI_COMMIT_BEFORE_SHA != "0000000000000000000000000000000000000000"
      changes:
        - $MODULE_PATH/**/*
        - parent-pom/**/*

Deployment Rollbacks

Resource Locking & Emergency Rollbacks

To avoid concurrent deployment race conditions, we use resource groups to serialize operations, paired with manual rollback fallback commands using standard kubectl rollout undo actions. 

deploy-manifests-prod:
  extends: .deploy-cluster-template
  stage: deploy
  resource_group: core-deploy-prod
  environment:
    name: prod
  script:
    - kubectl apply -k project/mmm-core/k8s/overlays/prod

rollback-prod:
  extends: .rollback-template
  stage: rollback
  needs: ["deploy-manifests-prod"]
  when: manual
  resource_group: core-deploy-prod
  variables:
    DEPLOYMENT_NAME: "mmm-core"
    K8S_NAMESPACE: "prod"
  script:
    - echo "Executing rollback deployment revert..."
    - kubectl rollout undo deployment/$DEPLOYMENT_NAME -n $K8S_NAMESPACE
Security Analysis

Shift-Left Security & SonarQube Project Splitting

By integrating automated SAST, secret detection, and dependency scanning early in developer workflow loops, security checks occur alongside active MR reviews. For SonarQube quality checks on multi-module structures, we configure isolated scanner profiles directly in each Maven module definition to prevent subsequent pipelines from overwriting metrics.

1. GitLab CI Templates Configuration (.gitlab-ci.yml) 
include:
  - template: Jobs/SAST.gitlab-ci.yml
  - template: Jobs/Secret-Detection.gitlab-ci.yml
  - template: Security/Dependency-Scanning.gitlab-ci.yml
2. Independent SonarQube Maven Properties (pom.xml) 
<properties>
    <sonar.version>5.5.0.6356</sonar.version>
    <sonar.projectKey>bwgjoseph:\${project.artifactId}</sonar.projectKey>
    <sonar.projectName>bwgjoseph:\${project.artifactId}</sonar.projectName>
    <sonar.organization>bwgjoseph</sonar.organization>
    <sonar.coverage.jacoco.xmlReportPaths>\${project.build.directory}/site/jacoco/jacoco.xml</sonar.coverage.jacoco.xmlReportPaths>

    <!-- SonarQube Scanner Configuration -->
    <sonar.scanner.skipSystemTruststore>true</sonar.scanner.skipSystemTruststore>
    <sonar.scanner.skipJreProvisioning>true</sonar.scanner.skipJreProvisioning>
    <!-- Java 25 support: Allow Sonar plugins to access restricted native methods -->
    <sonar.scanner.javaOpts>--enable-native-access=ALL-UNNAMED</sonar.scanner.javaOpts>
    <!-- Use CI project dir for scanner home if available, fallback to local .sonar -->
    <sonar.userHome>\${env.CI_PROJECT_DIR}/.sonar</sonar.userHome>
</properties>
